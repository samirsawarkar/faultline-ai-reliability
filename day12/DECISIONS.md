# DECISIONS.md — Day 12 (fault catalog + gallery) decision log

Append-only. Date · id · decision · why · reversal cost.

---

### 2026-07-22 · D12-001 · One normalized 10-field spec for all six families
**Decision.** Every fault card carries a spec with exactly the Day-8 10 field
names (`fault_id … spec_version`), regardless of family; `validate()` checks the
field set.
**Why.** "Consolidate every fault into a defensible specification" requires a
uniform shape. Reusing the Day-8 contract means the catalog speaks the same
language as the injector, and a diff of two cards is meaningful.
**Reversal cost.** None; it is the Day-8 contract, already frozen.

### 2026-07-22 · D12-002 · A card is invalid unless it has all five required fields
**Decision.** `FaultCard.validate()` raises unless trigger, trace, detector,
recovery, AND metric are all present and non-empty. `build_catalog` validates
every card, so an incomplete catalog cannot be built.
**Why.** This is the mission's fail condition, made mechanical. A checklist in
prose can rot; a validator that runs on every build cannot.
**Reversal cost.** None; it is the deliverable's guarantee.

### 2026-07-22 · D12-003 · Metrics are pulled live from the day that measured them
**Decision.** `catalog._metrics` reads Day 9/10/11 `build_report()` and copies the
observed recall/precision into each card, tagged with `measured_in`.
**Why.** A catalog of hand-typed numbers drifts from the code the moment either
changes. Pulling live keeps the card honest: if Day 10's F3 recall moves, the card
moves with it. It also means the catalog is not a new measurement, just a faithful
index of existing ones.
**Reversal cost.** None; re-derives from source reports.

### 2026-07-22 · D12-004 · Traces are regenerated by the owning day, not copied
**Decision.** `traces.canonical_trace` produces each fault's labelled trace using
the exact runner that owns the family (Day 9/10/11) plus the Day-4 tracer, and the
card references it by content digest.
**Why.** A trace pasted from an older evidence file could silently diverge from the
current code. Regenerating from the owner guarantees the catalog trace is the same
artifact the family's tests trust.
**Reversal cost.** Low; the producers are thin wrappers over public runners.

### 2026-07-22 · D12-005 · Recovery is recorded now, as a plan grounded in today's signal
**Decision.** Each card's `recovery` names the strategy, the signal already emitted
today (e.g. `repair_signal`, `timeout_signal`, circuit-breaker, `loop_detect`), and
the planned mechanism/mission — and whether recovery is deterministic.
**Why.** The mission requires a recovery field before the recovery missions
(18–22) are built. Grounding it in an already-emitted signal keeps it honest (not
vaporware) while pointing forward. For F3/F5 it truthfully says recovery is not
deterministic — the escape set needs a judge.
**Reversal cost.** Recovery missions will replace the plan with a measured
mechanism; the field's shape stays.

### 2026-07-22 · D12-006 · The audit is part of the catalog, not a side note
**Decision.** `run_audit` checks reproducibility (per fault + whole catalog),
ground-truth integrity (no label leakage in any trace), completeness (all five
fields), and that every metric is scored vs truth — and `audit_passed` gates them
all.
**Why.** A "defensible" specification must defend itself on demand. Shipping the
audit as committed evidence means anyone can re-run it and see the catalog holds.
**Reversal cost.** None; additive safety, reuses Days 8/9 discipline.

### 2026-07-22 · D12-007 · Taxonomy organized by producing component
**Decision.** The failure taxonomy groups faults by the component that produces
them (provider boundary vs agent internals; content vs transport vs state vs
control), not by symptom.
**Why.** Hardening is per-component. Organizing by producer turns the catalog into
an action plan and surfaces the pattern (transport/control faults deterministic;
data-plane and context value faults semantic) that Day 11's map already proved.
**Reversal cost.** None; it is a view over the same cards.
